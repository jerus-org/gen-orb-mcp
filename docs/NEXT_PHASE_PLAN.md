# Next Development Phase: Tier 2 Completion

## Intended End State

When this phase is complete, the gen-orb-mcp CircleCI orb will provide eight jobs:

| Job | Purpose |
|-----|---------|
| `prime` | Populate `prior-versions/` and `migrations/` from git history |
| `generate` | Generate MCP server source from orb YAML, with history embedded |
| `validate` | Validate an orb definition |
| `diff` | Compute conformance rules between two orb versions |
| `migrate` | Apply migration rules to a consumer `.circleci/` directory |
| `build` | Compile generated MCP server source to a native binary *(new)* |
| `publish` | Upload a compiled binary to an existing GitHub release *(new)* |
| `save` | Commit and push generated artifacts back to the repository *(new)* |

A consumer can wire these jobs into their CI workflow to achieve the full objective:

1. **Populate history** — `prime` discovers prior orb versions from git tags and writes
   `prior-versions/` snapshots and `migrations/` conformance rules.
2. **Generate** — `generate` produces MCP server source with history embedded.
3. **Compile** — `build` compiles the generated source to a binary.
4. **Publish** — `publish` uploads the binary to the GitHub release (which must already
   exist before this job runs).
5. **Persist** — `save` commits `prior-versions/`, `migrations/`, and any changed orb
   source back to the repository so the next CI run does not have to rebuild them from
   scratch.

Without step 5, `prime` must re-checkout every version tag in git history on every run —
a cost that grows with every release and becomes significant in long-running projects.

## Architecture Principle

The orb's job structure is directly determined by the CLI's subcommand structure.
`gen-circleci-orb generate` introspects `<binary> --help` and produces one orb command
file and one orb job file per subcommand. Workflow-level composition (jobs that combine
multiple commands) is deferred to a gen-circleci-orb configuration module (see
jerus-org/gen-circleci-orb#42). For this phase, every new capability is a CLI subcommand
so that gen-circleci-orb generates the corresponding orb job automatically.

## Tier 2 Additions

### 1. `build` subcommand

Compiles the generated MCP server source to a native binary.

```
gen-orb-mcp build [OPTIONS] --input <DIR>

Options:
  -i, --input <DIR>       Directory containing generated MCP server source [required]
  -o, --output <DIR>      Output directory for the compiled binary
                          [default: <input>/target/release]
  -n, --name <NAME>       Override the binary name (default: derived from Cargo.toml)
      --target <TRIPLE>   Rust target triple [default: host]
      --dry-run           Print the cargo command without running it
```

**What it does:**

1. Verifies `<input>/Cargo.toml` exists
2. Runs `cargo build --release` in `<input>` (or against the specified target)
3. Reports the binary path on success

Binary compilation can take 3-5 minutes without producing output. The generated orb job
sets `no_output_timeout: 15m`.

**Orb job generated:** `build` — wraps this subcommand. Required parameter: `input`.

---

### 2. `publish` subcommand

Uploads a compiled binary to an existing GitHub release as a release asset.

**Prerequisite:** The GitHub release for the tag must already exist before this job runs.
The subcommand does not create releases; it only uploads assets to an existing release.

```
gen-orb-mcp publish [OPTIONS] --binary <PATH> --asset-name <NAME>

Options:
  -b, --binary <PATH>     Path to the binary file to upload [required]
  -a, --asset-name <NAME> Name for the release asset [required]
  -t, --target <TARGET>   Distribution target [default: github-release]
                          Possible values: github-release
      --tag <TAG>         Release tag to publish to [default: $CIRCLE_TAG]
      --dry-run           Describe the upload without performing it
```

**Environment variables consumed (`github-release` target):**

| Variable | Purpose |
|----------|---------|
| `GITHUB_TOKEN` | Bearer token with `contents: write` permission |
| `CIRCLE_PROJECT_USERNAME` | Repository owner |
| `CIRCLE_PROJECT_REPONAME` | Repository name |
| `CIRCLE_TAG` | Release tag (overridden by `--tag`) |

**Implementation:** Uses `reqwest` with `rustls` (no native TLS), matching the approach
used in `pcu` (`reqwest = { version = "0.13.2", default-features = false, features = ["rustls"] }`).
No dependency on the `gh` CLI or any external tool.

**What it does:**

1. Resolves the release ID for the tag via the GitHub Releases API
2. Uploads the binary as a release asset via the uploads API
3. Reports the asset URL on success, or a diagnostic message if the release is not found

**Orb job generated:** `publish` — wraps this subcommand. Required parameters: `binary`,
`asset_name`. Context providing `GITHUB_TOKEN` required in the workflow entry.

### 3. `save` subcommand

Commits staged artifacts back to the repository and pushes, so they are preserved for
the next CI run.

Tracked as: jerus-org/gen-orb-mcp#118

```
gen-orb-mcp save [OPTIONS] <PATHS>...

Arguments:
  <PATHS>...        Paths to stage and commit

Options:
  -m, --message <MSG>   Commit message
                        [default: "chore: update generated MCP server artifacts [skip ci]"]
      --push            Push after committing [default: true]
      --no-push         Stage and commit only, do not push
      --dry-run         Show what would be committed without writing anything
```

**Idempotency:** If the working tree is clean after staging the specified paths, the
subcommand exits successfully without creating an empty commit.

**Implementation:** Uses the same git and GitHub API approach as the `pcu` library to
ensure consistent authentication, branch protection bypass authority, and push behaviour
across the jerus-org toolchain. A dedicated git-focused CLI built on the pcu library
would be a cleaner long-term solution and keep git operations out of gen-orb-mcp's scope;
that is a future direction (requires a new crate in pcu) and is not part of this phase.

**`[skip ci]`** in the default commit message prevents CI from re-triggering on the
artifact commit.

**Orb job generated:** `save` — wraps this subcommand.

## Complete Workflow After Tier 2

The following shows a release pipeline using all eight orb jobs. The workflow assumes
the GitHub release for `$CIRCLE_TAG` has been created by an earlier job before `publish`
runs. Filters and most contexts are omitted for clarity.

```yaml
orbs:
  gen-orb-mcp: jerus-org/gen-orb-mcp@<version>

workflows:
  release:
    jobs:
      # 1. Populate prior-version history from git tags
      - gen-orb-mcp/prime:
          orb_path: src/@orb.yml
          earliest_version: "1.0.0"
          ephemeral: true

      # 2. Generate MCP server source with embedded history
      - gen-orb-mcp/generate:
          requires: [gen-orb-mcp/prime]
          orb_path: src/@orb.yml
          output: /tmp/mcp-build
          version: "${CIRCLE_TAG#v}"
          migrations: /tmp/gen-orb-mcp-prime/migrations
          prior_versions: /tmp/gen-orb-mcp-prime/prior-versions

      # 3. Compile generated source to binary
      - gen-orb-mcp/build:
          requires: [gen-orb-mcp/generate]
          input: /tmp/mcp-build

      # 4. Upload binary to the already-created GitHub release
      - gen-orb-mcp/publish:
          requires: [gen-orb-mcp/build]
          binary: /tmp/mcp-build/target/release/my_orb_mcp
          asset_name: my-orb-mcp-linux-x86_64
          context: github-release   # must provide GITHUB_TOKEN

      # 5. Commit generated artifacts so next run does not rebuild from scratch
      - gen-orb-mcp/save:
          requires: [gen-orb-mcp/generate]
          paths: prior-versions migrations orb/src
          context: github-push      # must provide push credentials
```

Steps 3, 4, and 5 are what tier 2 adds. Steps 1 and 2 already work.

## Implementation Plan

### Approach: TDD throughout

For each new subcommand: write failing tests first (RED), implement to pass (GREEN),
verify no regressions.

### Phase A — `build` subcommand

1. Add `Build` variant to `Commands` enum in `lib.rs`
2. Create `crates/gen-orb-mcp/src/commands/build.rs`
3. RED: write unit tests for:
   - `cargo build --release` is invoked in the input directory
   - Binary output path is reported correctly
   - Missing `Cargo.toml` returns a clear error
   - `--dry-run` prints the cargo command without running it
4. GREEN: implement `run()` to pass tests
5. Update trycmd snapshot: `tests/cmd/build.trycmd`
6. Run gen-circleci-orb against updated binary to regenerate `orb/src/`

### Phase B — `publish` subcommand

1. Add `Publish` variant to `Commands` enum
2. Add `reqwest` to workspace `Cargo.toml` following pcu's pattern:
   `reqwest = { version = "0.13.2", default-features = false, features = ["rustls"] }`
3. Create `crates/gen-orb-mcp/src/commands/publish.rs`
4. Create `crates/gen-orb-mcp/src/github/mod.rs` for GitHub Releases API client
5. RED: write unit tests for:
   - `--dry-run` prints upload parameters without calling the API
   - Missing `GITHUB_TOKEN` returns a clear error before any API call
   - Release not found returns a diagnostic with the tag and repo slug
   - Successful upload logs the asset URL
6. GREEN: implement `run()` and `github::releases` client
7. Update trycmd snapshot
8. Run gen-circleci-orb to regenerate `orb/src/`

### Phase C — `save` subcommand

4. RED: write unit tests for:
   - Clean working tree after staging exits 0 without creating a commit
   - Changed paths produce a commit with the default message
   - `--dry-run` lists staged changes without writing
   - `--no-push` commits but does not push
5. GREEN: implement `run()`
6. Update trycmd snapshot
7. Run gen-circleci-orb to regenerate `orb/src/`

## Out of Scope for This Phase

- Cross-compilation (macOS, Windows targets) — Linux x86_64 only
- Additional `publish` targets (S3, GitLab releases) — `github-release` only
- Workflow-level composed jobs (`sync`, `build-server`) — deferred to
  jerus-org/gen-circleci-orb#42
- Dedicated git CLI built on pcu library — future direction, requires new pcu crate
