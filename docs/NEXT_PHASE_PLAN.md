# Next Development Phase: Tier 2 Completion

## Summary

This document covers the next phase of gen-orb-mcp development. It describes what has
been implemented, what is missing, and the plan to close the gaps.

---

## Current State

### CLI subcommands (implemented)

| Subcommand | Purpose |
|------------|---------|
| `generate` | Parse orb YAML and emit MCP server source |
| `validate` | Validate an orb definition |
| `diff` | Compute conformance rules between two orb versions |
| `migrate` | Apply conformance rules to a consumer `.circleci/` directory |
| `prime` | Populate `prior-versions/` and `migrations/` from git history |

### CircleCI orb (generated by gen-circleci-orb, implemented)

The orb `jerus-org/gen-orb-mcp` is published and exposes each subcommand as a job. Source
lives in `orb/src/` and is regenerated from the binary on every build.

### Gaps in the current workflow

A consumer who wants to ship a compiled MCP server binary alongside their orb release
cannot do so using only the current orb jobs. Two operations are missing from the CLI (and
therefore from the generated orb):

1. **Compile** â€” no `build` subcommand, so no generated orb job for turning source into a binary
2. **Distribute** â€” no `publish` subcommand, so no generated orb job for uploading the binary
   to a release or artifact store

A third gap affects artifact preservation between CI runs:

3. **Commit and push** â€” prime artifacts (`prior-versions/`, `migrations/`) and regenerated
   orb source are ephemeral unless committed back to the repository. Without this, every CI
   run re-derives everything from scratch. This is discussed separately below.

---

## Architecture Principle

The orb's job structure is directly determined by the CLI's subcommand structure.
`gen-circleci-orb generate` introspects `<binary> --help` and produces one orb command
file and one orb job file per subcommand. Workflow-level composition (jobs that combine
multiple commands) is deferred to a gen-circleci-orb configuration module (see
jerus-org/gen-circleci-orb#<issue>). For this phase, every new capability must be a CLI
subcommand so that gen-circleci-orb can generate it.

## Tier 2 Additions

### 1. `build` subcommand

Compiles the generated MCP server source to a native binary.

```
gen-orb-mcp build [OPTIONS] --input <DIR>

Options:
  -i, --input <DIR>       Directory containing generated MCP server source [required]
  -o, --output <DIR>      Output directory for the compiled binary [default: <input>/target/release]
  -n, --name <NAME>       Override the binary name (default: derived from orb name)
      --target <TRIPLE>   Rust target triple [default: host]
      --dry-run           Print what would be compiled without running cargo
```

**What it does:**

1. Verifies `<input>/Cargo.toml` exists
2. Runs `cargo build --release` in `<input>` (with appropriate target)
3. Reports the binary path on success

The compilation step can take 3-5 minutes without producing output. The subcommand must
handle `no_output_timeout` gracefully (the orb job will set `no_output_timeout: 15m`).

**Orb job generated:** `build` â€” wraps this subcommand. Required parameters: `input`.

---

### 2. `publish` subcommand

Distributes a binary artifact. The first implementation targets GitHub releases; additional
targets can be added without changing the interface.

```
gen-orb-mcp publish [OPTIONS] --binary <PATH> --asset-name <NAME>

Options:
  -b, --binary <PATH>     Path to the binary file to publish [required]
  -a, --asset-name <NAME> Name for the published asset [required]
  -t, --target <TARGET>   Distribution target [default: github-release]
                          Possible values: github-release
      --tag <TAG>         Release tag to publish to [default: $CIRCLE_TAG]
      --dry-run           Describe the upload without performing it
```

**Environment variables consumed:**

| Target | Variable | Purpose |
|--------|----------|---------|
| `github-release` | `GITHUB_TOKEN` | Bearer token with `contents: write` |
| `github-release` | `CIRCLE_PROJECT_USERNAME` | Repo owner |
| `github-release` | `CIRCLE_PROJECT_REPONAME` | Repo name |
| `github-release` | `CIRCLE_TAG` | Release tag (overridden by `--tag`) |

**What the `github-release` implementation does:**

1. Looks up the GitHub release for the tag via the Releases API
2. Uploads the binary as a release asset using the uploads API
3. Reports the asset URL on success

No dependency on `gh` CLI or any external tool. Uses `reqwest` (already a dependency).

**Orb job generated:** `publish` â€” wraps this subcommand. Required parameters: `binary`,
`asset_name`. Optional: `target`, `tag`.

## The Commit/Push Gap

Committing generated artifacts back to the repository is essential for preserving:

- `prior-versions/*.yml` â€” orb snapshots discovered by `prime`
- `migrations/*.json` â€” conformance rules computed by `diff`
- `orb/src/` â€” orb source regenerated by `generate` when CLI parameters change

Without a commit step, every CI run starts from scratch. `prime` must re-checkout every
version tag from git history on each run, which is slow and redundant. More critically,
migration rules computed during a release are lost if not committed.

### Why `commit` is not a gen-orb-mcp CLI subcommand

Committing and pushing is a git operation, not an MCP server generation operation. Adding
it to gen-orb-mcp would couple the tool to CI infrastructure and widen its scope beyond
its purpose.

The correct home is the gen-circleci-orb **configuration module** (see
jerus-org/gen-circleci-orb#42), which will
allow orb jobs to be defined that are not 1:1 mappings of a CLI subcommand. A `commit`
job defined in the configuration module would:

```yaml
- run:
    name: Commit generated artifacts
    command: |
      git config user.email "${GIT_AUTHOR_EMAIL}"
      git config user.name "${GIT_AUTHOR_NAME}"
      git add prior-versions/ migrations/ orb/src/
      git diff --cached --quiet || \
        git commit -m "chore: update generated MCP server artifacts [skip ci]"
      git push origin "${CIRCLE_BRANCH}"
```

### Interim approach

Until the gen-circleci-orb configuration module exists, consumers who need to commit
artifacts can:

1. Add a raw `run:` step in their workflow after the `prime` and `generate` jobs
2. Use `toolkit/commit_and_push` if they use circleci-toolkit

This is explicitly documented as an interim gap, not a supported pattern.

## Complete Workflow After Tier 2

A full release workflow using the orb (filters and contexts omitted for clarity):

```yaml
orbs:
  gen-orb-mcp: jerus-org/gen-orb-mcp@<version>

workflows:
  release:
    jobs:
      # 1. Populate prior-version history from git tags
      - gen-orb-mcp/prime:
          orb_path: src/@orb.yml
          earliest_version: "1.0.0"
          ephemeral: true

      # 2. Generate MCP server source with embedded history
      - gen-orb-mcp/generate:
          requires: [gen-orb-mcp/prime]
          orb_path: src/@orb.yml
          output: /tmp/mcp-build
          version: "${CIRCLE_TAG#v}"
          migrations: /tmp/gen-orb-mcp-prime/migrations
          prior_versions: /tmp/gen-orb-mcp-prime/prior-versions

      # 3. Compile generated source to binary
      - gen-orb-mcp/build:
          requires: [gen-orb-mcp/generate]
          input: /tmp/mcp-build

      # 4. Upload binary to GitHub release
      - gen-orb-mcp/publish:
          requires: [gen-orb-mcp/build]
          binary: /tmp/mcp-build/target/release/my_orb_mcp
          asset_name: my-orb-mcp-linux-x86_64
          context: github-release
```

Steps 3 and 4 are what tier 2 adds. Steps 1 and 2 already work.

## Implementation Plan

### Approach: TDD throughout

For each new subcommand: write failing tests first (RED), implement to pass (GREEN),
verify no regressions.

### Phase A â€” `build` subcommand

### Phase B â€” `publish` subcommand

   - `--dry-run` prints upload parameters without calling API
   - Missing `GITHUB_TOKEN` returns a clear error before any API call
   - Release not found returns a diagnostic message with the tag and repo slug
   - Successful upload logs the asset URL
5. GREEN: implement `run()` using `reqwest` (already in workspace dependencies)
6. Update trycmd snapshot
7. Run gen-circleci-orb against updated binary to regenerate `orb/src/`
